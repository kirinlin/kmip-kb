# Process

## Purpose

`Process` applies a cryptographic operation — encryption, decryption, signing, verification, key wrapping, key derivation, or another server-supported algorithm action — to supplied data, using a managed key stored on the server. The key material is never returned to the client. The server performs the computation and returns only the result, keeping the key confined to the server boundary at all times.

`Process` generalizes the pre-v2.1 [`MAC`](mac.md) operation to cover a broader set of cryptographic functions. It was introduced in v2.1 as the primary mechanism for server-side cryptographic operations.

## Request Fields

| Field | Tag | XML Text | Required | Description |
|---|---|---|---|---|
| Unique Identifier | `420094` | `UniqueIdentifier` | No | Identifies the managed key to use. Defaults to the server's ID Placeholder when omitted. The referenced object must be in a state that permits the requested cryptographic use. |
| Cryptographic Parameters | `42002B` | `CryptographicParameters` | No | Specifies the algorithm, cipher mode, padding scheme, hash function, and other parameters governing how the operation is performed. When omitted, the server uses the key's default Cryptographic Parameters attribute. |
| Data | `4200C2` | `Data` | No | The input data for the operation. | Init Indicator | `4200D7` | `InitIndicator` | No | When `True`, signals that this is the first request in a multi-part operation. The server allocates state for the operation and returns a handle. |
| Final Indicator | `4200D8` | `FinalIndicator` | No | When `True`, signals that this is the last request in a multi-part operation, and the server should finalize and return the complete result. |
| Correlation Value | `4200D6` | `CorrelationValue` | No | Client-chosen value to correlate multi-part operation continuations with the originating request. |
| Additional Authentication Data |  |  | No | For authenticated encryption modes (e.g., AES-GCM), the additional data to be authenticated but not encrypted. |

## Behavior & Server Requirements

The server resolves the key identified by the Unique Identifier (or ID Placeholder), verifies that the client has permission to use the key for the requested operation type, confirms that the key is in an appropriate state (typically Active), and then performs the operation.

For multi-part operations — processing large data in chunks — the client sends multiple requests. The first carries `Init Indicator: True`; intermediate requests carry neither indicator; the last carries `Final Indicator: True`. The server maintains per-operation state between chunks and returns final output (and authentication tags, for AEAD modes) with the final response.

The Cryptographic Parameters must be compatible with the key type. If no parameters are supplied and the key has no default parameters, the server returns an error.

The server must enforce [Cryptographic Usage Mask](../attributes/cryptographic-usage-mask.md) restrictions. A key whose usage mask excludes the Encrypt bit cannot be used for encryption via `Process`, even if the client explicitly requests it.

## Errors

Uses centralized error handling per the [error handling](../concepts/error-handling.md) conventions. Common failure causes include:

- The key is not in the Active state — it may be Pre-Active, Deactivated, or Destroyed.
- The key's Cryptographic Usage Mask does not permit the requested operation.
- The Cryptographic Parameters are incompatible with the key type or the requested operation.
- The input data is the wrong size (e.g., not a multiple of the block size for a non-padded cipher).
- Authentication tag verification failed during authenticated decryption.
- The Unique Identifier refers to an object that does not exist.

## Examples

A client needs to encrypt a 256-byte payload using an AES-256-GCM key managed by the server without ever receiving the key material:

```
Operation: Process
  Unique Identifier: "aes256-data-key-7"
  Cryptographic Parameters:
    Cryptographic Algorithm: AES
    Block Cipher Mode: GCM
    Tag Length: 128
  Data: <256 bytes of plaintext>
  Additional Authentication Data: <context string>
```

The server responds with:

```
Data: <ciphertext>
IV/Counter/Nonce: <server-generated 96-bit IV>
Authentication Tag: <128-bit tag>
```

The client stores the ciphertext, IV, and tag for later decryption. To decrypt, it calls `Process` again with the same key identifier and parameters, supplying the ciphertext, IV, and tag.
