# Basic Asymmetric Key Store Server KMIP Profile

## Overview

The Basic Asymmetric Key Store Server KMIP Profile defines server requirements for storing and retrieving asymmetric key pairs in KMIP v1.1 using the [Basic Authentication Suite](basic-authentication-suite.md). The "Store" designation means the server accepts client-provided key material via Register rather than generating keys itself — contrast with the [Foundry variant](basic-asymmetric-key-foundry-and-server-kmip-profile.md), which adds Create Key Pair. RSA is the mandatory asymmetric algorithm.

The companion client profile is the [Basic Asymmetric Key Store Client KMIP Profile](basic-asymmetric-key-store-client-kmip-profile.md). For the TLS 1.2 variant, see [Asymmetric Key Store Server TLS 1.2 Authentication KMIP Profile](asymmetric-key-store-server-tls-1-2-authentication-kmip-profile.md).

## Required Operations and Objects

The server must support Register, Get, Get Attributes, Locate, Activate, Revoke, and Destroy on Public Key and Private Key objects. RSA is the mandatory algorithm; key formats must include PKCS#1, PKCS#8, and Transparent RSA key formats.

Required attributes: `Unique Identifier`, `Object Type`, `Cryptographic Algorithm`, `Cryptographic Length`, `Cryptographic Usage Mask`, `State`, `Initial Date`, `Activation Date`, `Deactivation Date`.

## Implications for Implementers

- The Store profile is appropriate when asymmetric keys are generated by an external process (an HSM, a CA, an enrollment workflow) and then imported into KMIP for lifecycle management and controlled distribution.
- Distinguish Private Key and Public Key object permissions: clients that need only the public portion for verification or encryption should not be granted access to Private Key Get operations.
- For server-side key generation, use the [Basic Asymmetric Key Foundry and Server profile](basic-asymmetric-key-foundry-and-server-kmip-profile.md).
