# Credential Type Enumeration

## Overview

The Credential Type enumeration identifies the kind of authentication credential the client presents in the Authentication field of a KMIP request header. KMIP supports multiple authentication models because key management spans a wide range of deployment contexts — from simple username/password web services to hardware-attested storage devices and session-ticket-based workflows. The server uses the credential type to route the authentication material to the appropriate validation mechanism.

## Fields & Structure

| Value | Hex | XML Text | Description |
|---|---|---|---|
| Username and Password | `0x00000001` | `UsernameAndPassword` |  |
| Device | `0x00000002` | `Device` |  |
| Attestation | `0x00000003` | `Attestation` |  |
| One Time Password | `0x00000004` | `OneTimePassword` |  |
| Hashed Password | `0x00000005` | `HashedPassword` |  |
| Ticket | `0x00000006` | `Ticket` |  |

- **Username and Password**: The simplest credential — a plaintext username and password pair transmitted over an already-secured transport (typically TLS). Widely supported but provides no hardware binding or replay protection beyond the transport layer.
- **Device**: A device-level credential that identifies a specific physical endpoint rather than a user, typically using a device certificate or identifier. Used for machine-to-machine authentication in storage and HSM integrations.
- **Attestation**: An attestation proof (TPM quote, TCG report, or SAML assertion) that demonstrates the client is running a verified software or hardware configuration. See [Attestation Type Enumeration](attestation-type-enumeration.md) for sub-type details.
- **One Time Password**: A short-lived password generated by a hardware or software token (TOTP/HOTP). Provides resistance to replay attacks because each code is valid only for a brief window.
- **Hashed Password**: A pre-hashed form of a password, allowing the server to verify credentials without requiring the client to transmit cleartext. Useful in legacy protocol bridging scenarios.
- **Ticket**: A session ticket issued by the KMIP server (via the [Login](../operations/login.md) operation) that the client presents in subsequent requests to avoid re-supplying full credentials. See [Ticket Type Enumeration](ticket-type-enumeration.md).

## Examples

A database application authenticating to KMIP with a service account uses **Username and Password** over mTLS. A self-encrypting drive controller uses **Device** credentials based on its factory certificate. A cloud workload running in a verified container uses **Attestation** to prove its runtime environment before receiving encryption keys.
