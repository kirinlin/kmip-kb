# Basic Symmetric Key Store and Server KMIP Profile

## Overview

The Basic Symmetric Key Store and Server KMIP Profile defines the minimum requirements for a KMIP server that stores client-provided symmetric keys, along with the corresponding client interaction model. The "Store" designation distinguishes this profile from the [Foundry variant](basic-symmetric-key-foundry-and-server-kmip-profile.md): the server receives and holds key material supplied by clients via Register rather than generating keys itself. This profile uses the [Basic Authentication Suite](basic-authentication-suite.md); for the TLS 1.2 variant see [Symmetric Key Store and Server TLS 1.2 Authentication KMIP Profile](symmetric-key-store-and-server-tls-1-2-authentication-kmip-profile.md).

This profile first appeared in KMIP-Prof v1.0 as a combined client-and-server definition. KMIP-Prof v1.1 retained it and added a dedicated [Basic Symmetric Key Store Client](basic-symmetric-key-store-client-kmip-profile.md) profile for clients that need only declare their side of the interaction.

## Required Operations and Objects

**Server** — must support Register, Get, Get Attributes, Get Attribute List, Locate, Activate, Revoke, Destroy, and Destroy on Symmetric Key objects. AES is the mandatory symmetric algorithm; key format support must include Raw and Transparent Symmetric Key.

**Client** — must be able to invoke Register to store a key and Get to retrieve it. Additional operations (Locate, Activate, Revoke, Destroy) are expected to be called as needed.

Core attributes required: `Unique Identifier`, `Object Type`, `Cryptographic Algorithm`, `Cryptographic Length`, `Cryptographic Usage Mask`, `Initial Date`, `State`, `Activation Date`, `Deactivation Date`.

## Implications for Implementers

- The Store profile suits deployments where key material is generated or derived outside KMIP and then deposited at the server for lifecycle management and controlled access — for example, keys generated by an HSM that are registered into KMIP for audit and distribution.
- AES is mandatory. 3DES support is optional but useful for legacy interoperability.
- Ensure that Register operations validate the incoming key format and set the object state correctly to `Pre-Active` at registration.
- If your environment requires mutual TLS 1.2, target the TLS 1.2 variant instead of this profile.
